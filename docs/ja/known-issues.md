# 既知の問題

現時点の制限事項の一覧。ルーティング規則の詳細は [device-model.md](device-model.md) を参照。

## CH → FX send の PRE/POST は実機へ反映できない

チャンネルから **FX 1 / FX 2** への send の PRE/POST は、プランナー上では自由に
編集でき (プランが意図する値を保持します)、software から URX へは書き込めません
(この設定は本体の操作パネル (LCD) からのみ変更できます)。そのため Live sync 接続中は
コントロールが読み取り専用 (disabled・マウスオーバーで注記) になり、値は実機の状態を
表示します (readback が最新に保ちます)。これはインスペクタの PRE/POST トグルと
CONSOLE ビューの PRE ボタンの両方に適用されます。オフライン (純粋なプランナー) では
編集可能なままです。

**CH → MIX** および **FX チャンネル → MIX** send の PRE/POST は従来どおり実機へ
反映できます。

> 背景: CH → FX send の PRE/POST は本体の操作パネルでのみ設定でき (broker が
> software からの書き込みを拒否する)、アプリは値を readback するため Live 中は
> 常に実機の真の値を表示します。

## CH SETTING の Icon は非対応

実機の CH SETTING には名前・色と並んで **Icon** の設定項目がありますが、プランナーでは
意図的に非対応としています。mono チャンネル (CH1–4) は実機がアイコンを broker に公開
しないため制御できず、stereo チャンネルとバスのみ公開する非対称な機能になってしまうため
です。名前 (`nodeNames`) と色 (`nodeColors`) は全ノードで読み書きできるため対応して
います。

## CUE (ソロ/モニタ割り込み) の割当は制御できない

実機の各チャンネルは **CUE** ボタンを持ち、押すと CUE バス経由でそのチャンネルを
モニタへソロ的に割り込ませます。プランナーではこの **CUE バスへの割当 (どのチャンネルを
CUE するか) を表現せず、実機へも反映しません**。CUE のルーティングは電源 OFF で削除される
一時的なバスであり、保存する計画として永続的な割当を保持できないためです
([device-model.md](device-model.md) 参照)。

MONITOR バスの **CUE Int** (CUE 割り込みの有効/無効、`MONITOR_CUE_INTERRUPT`) は
確定パラメータのため、CONSOLE の MONITOR ストリップとインスペクタで読み書きでき、
Live sync 中も同期します。制御できないのは個々のチャンネルの CUE オン/オフ (割当) です。

## 実機制御の検証は URX44V のみ

実機制御は実機の **URX44V** で開発・検証しています。**URX44** は URX44V の制御
マップをそのまま流用しており (ハードウェア差は HDMI 入力のみで、既定ではルーティング
されません)、一致する想定ですが実機での検証はしていません。**URX22** は、CONSOLE の
ライブメーターのルーティングを URX22 オーナーが実機で確認済みです (ステレオチャンネルの
メーターはステレオペアの位置でインデックスされ、URX22 は mono チャンネルが 2 本のみのため
先頭ステレオチャンネルが CH3/4 になります)。ただし制御 (書き込み) マップと工場初期プランは
依然 URX44V からの類推で未検証のため、値が実機と完全には一致しない可能性があります。
オフラインのプラン作成・プラン JSON・画像
出力には影響せず、対象はこの2機種の Live sync のみです。

**動作確認環境。** 実機制御は以下の組み合わせで確認しています。将来のファームウェアや
Device Center のアップデートで制御プロトコルが変更される可能性があるため、新しい
バージョンでの同一動作は保証されません。

| 構成要素 | 確認バージョン |
| --- | --- |
| 機種 | URX44V |
| ファームウェア | V1.3.0.1 |
| Device Center | 2.2.0 (2.2.0.2) |

接続時に実機の System ファームウェアバージョン (`/vd/device`) を読み取り、上表の
確認バージョンと異なる場合は読込・書込・Live sync の開始時に「正しく動作しない
可能性がある」旨を警告します。続行するか中断するかはユーザーが選択でき、強制的な
停止はしません (ファームウェアを読み取れない場合は警告を出さず続行します)。

## AUTO (オートゲイン) トリガーは非対応

実機の入力画面には、入力ゲインのワンショット自動測定を実行する **AUTO** ボタンが
あります。プランナーではこれを非対応としています。これは保存される設定ではなく
ライブのアクションであり、保存プランやスナップショット差分で表現する対象が無いため
です。入力ゲイン自体は従来どおりプラン・同期できます。対象外なのは自動測定の
トリガーのみです。

## SD Rec の Track Count は read-only

microSD レコーダーのトラック毎ソース割当は編集でき、Live sync で同期されます
(各ステレオトラックペアがソース — チャンネルペア・STEREO・MIX バス — を選択・
param 736。[device-model.md](device-model.md) 参照)。

一方、レコーダーの **Track Count** (2 … 16) は **read-only** です。CH → FX send の
PRE/POST と同様、実機は software 書込みを受理しても無視し、本体の操作パネルからのみ
変更できます (param 839)。プランナーは readback で表示し、表示するトラックペア
スロット数の制御に使いますが実機へは push できません — 保存したプランの Track Count
は実機へ書き込まれません。

これは URX44 / URX44V のみです (URX22 に microSD 録音はありません)。

## STREAMING の pre-DELAY メーターは取得できない

ブロックダイアグラムでは STREAMING チャンネルに DELAY の前後 2 つのメーターが
描かれていますが、実機ブローカーが公開するのは **post-DELAY** メーターのみで、
**pre-DELAY** メーターはどの状態でもデータを返しません (実機 URX44V で確認)。
そのため CONSOLE の STREAMING ストリップは post-DELAY (出力) メーターのみを
表示し、メーターポイントのセレクタは持ちません。ディレイを設定すると pre/post で
レベル変化のタイミングは確かに異なりますが、実機が pre-DELAY 値を出さないため
表示できません — pre-DELAY レベルに最も近いのは、STREAMING のソースとなる
バス (STEREO / MIX) 自身のメーターです。

## サンプルレートは実機の Follow USB が OFF のときだけ書き込まれます

プランナーの **レート** 設定は実機へ書き込まれ (param 766)、デバイスを再クロック
します。この書き込みが効くのは、実機側の **Follow USB** が **OFF** のときだけです。
Follow USB が ON の場合、URX は USB ホストのクロックに追従するため、ソフトウェアからの
レート書き込みは無視されるか即座に上書きされます。

Follow USB 自体はソフトウェアから読み書きできません。ブローカーのパラメーターとして
公開されていないためです (実機 URX44V で OFF → ON の切り替え前後の全パラメーター
スナップショットを差分し、変化するパラメーターが 1 つも無いことを確認済み)。したがって
プランナーは Follow USB を切り替えることも状態を検知することもできず、レート書き込みが
無視される状況を事前に警告できません。

運用上は、レート変更が実機に反映されない場合、デバイス側で **Follow USB を OFF** にして
(Setup → サンプルレート画面) からレートを設定し直してください。どちらの場合でも、プラン
自体は設定したレートを保持します。

## HDMI のサンプルレート上限は音声モードで変わる

HDMI 入力のサンプルレート上限は、デバイスの HDMI メニューで設定するモードに依存します。
**2ch モード**は 48 kHz 上限で、**Multi Channels モード**は 192 kHz まで対応し、
マルチチャンネル音声をステレオペアへ 8→2 で down-mix します。どちらのモードかは — したがって
上限も down-mix も — 入力される HDMI 信号に実行時で従い、保存プランが保持する情報では
決まらないため、プランナーはこの相互作用をモデル化しません (サンプルレート警告にも反映しません)。
HDMI 入力は選択可能なチャンネルソースのままで、8→2 down-mix はルーティングに現れます
([device-model.md](device-model.md) 参照)。モード依存のレート上限だけがスコープ外です。
